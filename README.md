# RTO SPC Dashboard for Semiconductor Tool Health Monitoring

A synthetic semiconductor manufacturing analytics dashboard for RTO tool-health monitoring, combining module-level thickness monitoring, particle-adder alerting, deterministic SPC logic, across-fleet visualization, and traceable excursion review.

---

## Overview

This project demonstrates how synthetic RTO monitor data can be transformed into an engineering-oriented tool-health dashboard using Python, Streamlit, Plotly, pandas, and pytest.

The dashboard is designed around two core RTO monitor categories:

### Thickness monitoring

* RTR Mean
* X-BAR
* WIW Stdev
* SIGMA

### Particle monitoring

* Total Adder
* Cluster Adder
* Large Particle Adder

The system calculates control limits from baseline data only, applies deterministic warning and OOC rules to monitoring data, and writes every detected event to a centralized excursion-event table.

---

## Why This Project Exists

RTO processes require tight process monitoring because relatively small thickness or uniformity changes can indicate meaningful changes in recipe stability, chamber condition, thermal behavior, metrology consistency, or contamination risk.

The project was created to demonstrate:

* Semiconductor process-monitoring knowledge
* RTO module-level engineering logic
* Statistical Process Control implementation
* Particle-monitor alert design
* Fleet-level tool and chamber comparison
* Traceable excursion-event architecture
* Python analytics and dashboard development
* Testing, documentation, and confidentiality discipline

## Core Design Principles

### RTO-only scope

The project is intentionally limited to synthetic RTO tool-health monitoring.

It does not attempt to model:

* Furnace operations
* FDC sensor traces
* APC feedback control
* Real FAB systems

### Baseline-only control limits

All control limits are calculated only from rows where:

```text
phase == "baseline"
```

Monitoring data is evaluated against frozen baseline limits.

Monitoring excursions are never allowed to modify the limits used to detect them.

### Per-stream SPC calculations

Control limits are calculated independently for each monitor stream:

```text
tool_id + chamber_id + recipe_group + metric_name
```

Data from different tools, chambers, recipes, or metrics is never combined to calculate a shared SPC limit.

### Fleet charts are visualization-only

Across-fleet charts compare multiple RTO tools and chambers, but they do not calculate fleet-wide control limits.

Each displayed stream preserves its own:

* Baseline Mean
* Baseline Sigma
* Upper Control Limit (UCL)
* Control Limit (CL)
* Lower Control Limit(LCL)
* Warning Status
* Out-Of-Control (OOC) Status

### Deterministic event detection

SPC and particle-event decisions are generated by deterministic rules.

Local summaries and suggested review areas do not determine SPC status and do not replace engineering review.

## Thickness Monitor Methodology

The Phase 1 RTO thickness monitor uses two primary SPC chart types:

| SPC chart | Underlying monitor metric | Engineering purpose                                                                             |
| --------- | ------------------------- | ----------------------------------------------------------------------------------------------- |
| X-BAR     | RTR Mean                  | Tracks run-to-run average thickness behavior and detects thickness mean shift or drift          |
| SIGMA     | WIW Stdev                 | Tracks within-wafer thickness variation and detects uniformity or process-variation degradation |

In this project:

```text
RTR Mean is plotted on the X-BAR chart.
WIW Stdev is plotted on the SIGMA chart.
```

X-BAR and RTR Mean are not separate monitor metrics.

SIGMA and WIW Stdev are not separate monitor metrics.

The internal metric names are:

```text
thickness_rtr_mean
thickness_wiw_stdev
```

The dashboard must not create duplicate metric streams named:

```text
thickness_xbar
thickness_sigma
```

### X-BAR chart

The X-BAR chart displays the RTR Mean for each synthetic RTO thickness-monitor run.

It is used to review:

* Run-to-run mean thickness behavior
* Process centering
* Gradual thickness drift
* Sudden mean shift
* Tool and chamber matching
* Metrology trend consistency

### SIGMA chart

The SIGMA chart displays the WIW Stdev for each synthetic RTO thickness-monitor run.

It is used to review:

* Within-wafer thickness uniformity
* Increasing wafer-level variation
* Chamber thermal uniformity
* Wafer placement or process-distribution changes
* Metrology repeatability

### Baseline-only limits

Control limits for both chart types are calculated independently from baseline data for each:

```text
tool_id + chamber_id + recipe_group + metric_name
```

Monitoring observations are evaluated against frozen baseline limits.

Across-fleet charts are used for tool and chamber comparison only. They must not calculate a shared fleet-wide control limit.

## Particle Monitor Methodology

Phase 1 particle monitoring uses synthetic engineering thresholds and repeated-event escalation.

It does not use c-chart or u-chart logic as the primary alert method.

| Metric               | Internal metric name     | Warning | High |
| -------------------- | ------------------------ | ------: | ---: |
| Total Adder          | `particle_total_adder`   |      10 |   20 |
| Cluster Adder        | `particle_cluster_adder` |       3 |    6 |
| Large Particle Adder | `particle_large_adder`   |       1 |    3 |

These thresholds are synthetic demonstration values and are not production limits.

### Particle rules

| Rule                                 | Condition                                                                                       | Severity |
| ------------------------------------ | ----------------------------------------------------------------------------------------------- | -------- |
| `PARTICLE_THRESHOLD_WARNING`         | Value reaches warning threshold but remains below high threshold                                | Medium   |
| `PARTICLE_THRESHOLD_HIGH`            | Value reaches or exceeds high threshold                                                         | High     |
| `PARTICLE_REPEATED_EVENT_ESCALATION` | At least three warning-level observations occur within the latest ten points in the same stream | High     |

Repeated-event logic is evaluated independently within:

```text
tool_id + chamber_id + recipe_group + metric_name
```

## Dashboard Pages

### 1. Tool Health Summary

Provides an across-fleet view of:

* Synthetic tool-health scores
* Event counts by tool
* Event counts by chamber
* Severity distribution
* Latest high-severity events

> Synthetic demo health score, not a production disposition rule.

### 2. RTO Thickness Monitor

Provides fleet-level and selected-stream views for:

* RTR Mean
* X-BAR
* WIW Stdev
* SIGMA

The selected-stream view includes:

* Value trend
* UCL, CL, and LCL
* Warning markers
* OOC markers
* Associated event records

### 3. Particle Alerts

Provides fleet-level and selected-stream views for:

* Total Adder
* Cluster Adder
* Large Particle Adder

The selected-stream view includes:

* Particle trend
* Warning threshold
* High threshold
* Medium-severity markers
* High-severity markers
* Repeated-event escalation markers
* Associated event records

### 4. Excursion Review

Provides a centralized review workflow based on `excursion_events.csv`.

Views include:

* Full event table
* Event count by metric
* Event count by tool
* Event count by chamber
* Severity distribution
* Excursion timeline
* Filtered CSV export

### 5. Local Event Summary

Generates a deterministic event summary using stored SPC and threshold results.

The summary may include:

* Event ID
* Tool and chamber
* Monitor metric
* Triggered rule
* Severity
* Suggested review area

> This summary is generated from deterministic SPC / threshold results and does not replace engineering review.

## Installation

### 1. Clone the repository

```bash
git clone <repository-url>
cd rto-spc-dashboard
```

### 2. Create a virtual environment

Windows:

```bash
python -m venv .venv
.venv\Scripts\activate
```

macOS or Linux:

```bash
python -m venv .venv
source .venv/bin/activate
```

### 3. Install dependencies

```bash
pip install -r requirements.txt
```

Alternatively:

```bash
pip install -e .
```
